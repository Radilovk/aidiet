# Native Vector Protocol (NVP) v2.2
## Native Token Interface Layer between Human Intent and LLM Tokenization

**Статус:** Request for Comments (RFC)  
**Версия:** 2.2  
**Дата:** 16 Юни 2026  

---

### 1. Абстракция (Abstract)

Съвременните големи езикови модели (LLMs) са авторегресивни предсказващи машини, чиито механизми на внимание (Attention Heads) са оптимизирани за синтактичните структури на естествените езици и програмния код по време на техния pre-training. Конвенционалните методи за компресия на текст или използването на символи извън разпределението (Out-of-Distribution - OOD), като CJK или Private Use Area (PUA) Unicode блокове, водят до деградация на семантичния вектор, фрагментация на BPE токенизатора и увеличаване на халюцинациите.

**Native Vector Protocol (NVP) v2.2** решава тези проблеми чрез **Code-Space Hijacking** – пренасочване на логическите вектори на модела към високостабилни среди: Unix Shell (Pipe синтаксис) и Обектно-ориентирано програмиране (Dot notation). NVP v2.2 постига 3-15x компресия на токените, запазвайки 100% инвариантност на семантичния вектор и пълна съвместимост с базовите BPE токенизатори (`cl100k_base`, `o200k_base`), без нужда от fine-tuning.

---

### 2. Архитектурен преглед (Layered Architecture)

NVP v2.2 е структуриран в три функционални слоя, работещи в синхрон с изчислителната физика на трансформър архитектурите.

| Слой | Наименование | Механизъм | BPE Ефективност | Предвидимост на Attention |
| :--- | :--- | :--- | :--- | :--- |
| **L0** | Native Pipe | Unix Shell Pipe + OOP Dot Notation | Максимална (1 токен на оператор) | Абсолютна (In-Distribution) |
| **L1** | Nibble Macros | Hex-indexed указатели (`$0` - `$FF`) | Висока (1-2 токена за макрос) | Стабилна (Статични променливи) |
| **L2** | Masked Codebook | Динамично ремапване на указатели | Висока (Транспортен слой) | Ограничена (Изисква контекст) |

---

### 3. Layer 0: Functional Pipe Syntax & Grammar

Layer 0 елиминира лингвистичния overhead (съюзи, предлози, граматически времена), като налага строго еднопосочен, ляво-надясно (Left-to-Right) информационен поток, съвпадащ с авторегресивното генериране.

#### 3.1 Формална граматика (EBNF)

```ebnf
<nvp_stream>    ::= [<context_block>] "|" [<compute_block>] <pipeline> "|>" <output_target>
<context_block> ::= "@@" <k_v_pairs>
<k_v_pairs>     ::= <key> "=" <value> ("," <key> "=" <value>)*
<compute_block> ::= "[~" <reasoning_tokens> "~]"
<pipeline>      ::= <stage> ("|" <stage>)*
<stage>         ::= <agent> "." <method> "(" [<arguments>] ")"
<arguments>     ::= <value> ("," <value>)*
<output_target> ::= <identifier>
```

#### 3.2 Контролни оператори (1-Token Anchors)
Всички оператори в NVP v2.2 са избрани заради техния висок статистически интензитет в `cl100k_base` речника, което гарантира токенизация от точно 1 токен:
* `@@` : Глобален контекст (Инициализация на паметта).
* `|` : Unix Pipe оператор (Предаване на състоянието по веригата).
* `.` : Обектен сепаратор (Връзка Агент -> Метод).
* `()` : Изолация на аргументи и параметри.
* `|>` : Краен изходен възел (Форсиране на локализация/формат). *Забележка: В някои BPE конфигурации се токенизира като 2 токена (`|` и `>`), но запазва перфектна семантична кохерентност.*

#### 3.3 Примери за транслация

* **Естествен език (Български):** „Котката вчера донесе мъртва мишка на леглото.“
* **NVP v2.2 съобщение:** `@@t=yest | cat.bring(mse.dead) |> bed`
* **Токенизация:** 9 токена срещу ~25-30 токена на чист текст.

### 4. Изчислителен изолационен слой: Динамичен CoT (Compute Block)

Трансформър моделите изискват изчислително пространство (Token Runway), за да извършват логически разсъждения чрез Chain-of-Thought (CoT). Премахването на това пространство води до деградация на отговорите при комплексни задачи.

NVP v2.2 въвежда **Асиметрична изчислителна рамка** чрез динамичния блок `[~ ... ~]`.

#### 4.1 Механизъм на изпълнение
1. **Поток:** Клиентът изпраща NVP стрийм.
2. **Изчисление:** Моделът отваря оператора `[~` и генерира вътрешен неструктуриран монолог (разсъждения, валидация на параметри, логически стъпки). След завършване на изчислението, моделът затваря оператора с `~]`.
3. **Полезен товар:** Моделът генерира финалния компресиран NVP изход.
4. **Клиентски парсер:** Регулярен израз на клиента филтрира и изрязва всичко между `[~` и `~]`. Крайният потребител или агент получава чист, компресиран полезен товар.

#### 4.2 Пример за транслиция на обработката

**Вход (Потребител):**
```
@@task=risk_analysis | corp.evaluate(fin_data) |> report
```

**Суров изход от LLM:**
```
[~ Валидиране на финансови коефициенти. Ликвидност под прага. Изчисляване на вероятност от дефолт чрез формула на Алтман. Сегментиране на риска. ~] corp.status(alert) | risk.level(high) |> report
```

**Филтриран изход (към клиентската система):**
```
corp.status(alert) | risk.level(high) |> report
```

### 5. Layer 1 & 2: Hex-Nibble Pointer Address Space

За инуитивно in-context адресиране на дълги системни инструкции, темплейти или често срещани фрази, NVP v2.2 отхвърля нестабилните PUA символи и въвежда шестнадесетична (Hex-Nibble) адресация, използваща префикса `$`. Токенизаторите разпознават изолираните нибъли като единични токени.

#### 5.1 Структура на адресното пространство

* **Единичен нибъл (`$0` – `$F`):** 16 статични макроса с тегло точно 1 токен. Предназначени за глобални системни флагове и деструктивни команди.
* **Двоен нибъл (`$A0` – `$FF`):** 256 макроса с тегло 2 токена. Предназначени за бизнес логика, специфични за домейна системни роли или предефинирани грешки.

#### 5.2 Bootstrap Инициализация (Системен Промпт)

```
You are an NVP v2.2 Core Engine. Decode incoming streams and execute state transitions.
Hex-Nibble Address Space:
$0 = "Error: Missing input data. Terminate pipeline."
$1 = "Execute Game Theory optimization matrix."
$A1 = "Role: Senior Analytics Engine. Output language: Bulgarian."
```

#### 5.3 Приложение на L1/L2 слоевете

**Извикване на стрийм:**
```
@@usr=id123 | plan.generate(market) |> $A1 | $1
```
*Семантичен превод:* „Потребител 123 изисква генериране на пазарен план. Изпълни го в ролята на Senior Analytics Engine на български език, прилагайки матрица за оптимизация от Теория на игрите.“  
*Общо токени:* 11 токена за комплексна системна инструкция и бизнес логика.

### 7. Имплементация и Клиентска интеграция

За внедряването на NVP v2.2 в производствени среди, клиентските софтуерни агенти (Node.js, Python, Cloudflare Workers) трябва да имплементират следния базов пайплайн за обработка:

```python
import re

def parse_nvp_response(raw_llm_output: str) -> str:
    # 1. Елиминиране на изчислителния CoT блок преди изпращане към крайния слой
    clean_payload = re.sub(r'\[~.*?~\]', '', raw_llm_output).strip()
    return clean_payload

# Примерна интеграция
raw_output = "[~ Вычисление вектора... ~] sys.status(ok) |> render"
print(parse_nvp_response(raw_output)) # Изход: sys.status(ok) |> render
```

#### 7.1 Основни изисквания към инженерите
1. **Забрана за разделители с празни пространства:** Всички аргументи вътре в `()` трябва да се разделят само със запетая `,` без интервали, за да се избегне сплитване на токени.
2. **Капсулация:** Всеки нов агент или микросервиз трябва да притежава уникален обхват в рамките на нибъл адресите (`$A0-$BF` за Агент 1, `$C0-$DF` за Агент 2), предотвратявайки колайдване на дефинициите в споделената памет.
