# Система за нотификации — Ръководство за разработчици

## Какво е това и защо е направено така

Приложението AIDiet изпраща две нотификации на ден — сутрешна и вечерна — за да канят потребителя да отговори на гамификационните въпроси. Целта е простота и надеждност: нотификациите се планират **локално на устройството**, без да зависят от постоянна сървърна заявка.

---

## Архитектура

```
┌─────────────────────────────────────────────────────────────┐
│                     Устройство (клиент)                      │
│                                                               │
│   local-scheduler.js  →  GameNotifier.init()                 │
│          │                                                    │
│          ├──[APK / Android]──▶  Capacitor LocalNotifications │  ✅ Най-надежден
│          │                      (работи дори при затворено   │
│          │                       приложение)                  │
│          │                                                    │
│          └──[PWA / браузър]──▶  Service Worker postMessage   │  ⚠️ Само докато
│                                 → setTimeout в SW             │     браузърът е жив
└─────────────────────────────────────────────────────────────┘
         ▲
         │  GET /api/notification-config
         │  (max 1 път на 24 ч, само ако версията е по-нова)
         │
┌────────┴────────────────────┐
│  Cloudflare Worker (backend) │
│  KV key: 'notification-config'│
└─────────────────────────────┘
```

---

## Ключови файлове

| Файл | Роля |
|---|---|
| `local-scheduler.js` | Целият клиентски код за нотификации — `GameNotifier` обект |
| `sw.js` | Service Worker — приема `SCHEDULE_GAME_NOTIFICATIONS` postMessage и изпраща нотификацията чрез `setTimeout` |
| `plan.html` | Инициира `GameNotifier.init()` след разрешение; обработва `?action=morning_check` и `?action=evening_check` URL параметри |
| `worker.js` | Backend API — `GET/POST /api/notification-config` за admin конфигурация |

---

## Как работи стъпка по стъпка

### 1. Инициализация (`plan.html` → `GameNotifier.init()`)

При зареждане на `plan.html` (ако потребителят е дал разрешение за нотификации):

```
GameNotifier.init()
  ├── Открива контекст (APK vs PWA)
  ├── Синхронизира конфига от backend (max 1 път/24 ч)
  └── Планира нотификации за следващите 7 дни
```

### 2. Планиране

**APK (Capacitor):**
- Планира 14 нотификации (7 сутрешни + 7 вечерни) чрез `LocalNotifications.schedule()`
- Нотификациите оцеляват дори при затворено приложение

**PWA (браузър):**
- Изпраща `postMessage` до Service Worker с масив от `{ts, title, body, tag, url}`
- SW използва `setTimeout` за всяка нотификация
- Работи само докато браузърът е отворен и SW е жив

### 3. Докосване на нотификацията

Когато потребителят докосне нотификацията, приложението се отваря на:
- `plan.html?action=morning_check` — задейства сутрешния гамификационен въпрос
- `plan.html?action=evening_check` — задейства вечерния гамификационен въпрос

### 4. Backend config (само за admin)

Admin може да промени часовете и текстовете на нотификациите от `admin.html`.  
Промяната се записва в KV като `notification-config` с нова версия.  
При следващото отваряне на `plan.html` клиентът сравнява версията и, ако е по-нова, изтегля новата конфигурация.

## Конфигурация по подразбиране

```js
morningTime:  '07:00'
eveningTime:  '20:00'
morningTitle: 'Добро утро! 🌅'
morningBody:  'Как спахте тази нощ? Отговорете на сутрешния въпрос.'
eveningTitle: 'Добър вечер! 🌙'
eveningBody:  'Как мина денят? Отговорете на вечерните въпроси.'
```

Конфигурацията се кешира в `localStorage` под ключ `gameNotifierConfig`.

## Как да тествате (без конзола)

### За реален тест на нотификации:

1. **APK на Android** — инсталирайте `.apk` файла на телефон, разрешете нотификации при първо стартиране. Нотификациите ще дойдат в 07:00 и 20:00 без да е нужно приложението да е отворено.

2. **PWA в Chrome** — отворете `plan.html`, разрешете нотификациите при подкана. Нотификациите ще дойдат само докато браузърът е отворен (ограничение на платформата).

3. **Директен тест на сутрешния/вечерния въпрос** — отворете директно URL:
   - `https://aidiet.radilov-k.workers.dev/plan.html?action=morning_check`
   - `https://aidiet.radilov-k.workers.dev/plan.html?action=evening_check`

### Промяна на часовете (само за admin):

1. Влезте в `admin.html`
2. Намерете секцията за нотификации
3. Сменете часа и запазете — промяната ще се разпространи до клиентите при следващото им отваряне на приложението

## Защо е направено така (архитектурни решения)

| Решение | Причина |
|---|---|
| Локално планиране (не сървърен cron) | Не зависи от Cloudflare Workers cron лимити; работи офлайн |
| Capacitor за APK | Единственият надежден начин за нотификации при затворено Android приложение |
| SW setTimeout за PWA | Прост fallback без нужда от VAPID/push сървър за базовата функционалност |
| 24 ч throttle за backend sync | Намалява броя API заявки; admin конфигурацията рядко се сменя |
| 7-дневен блок нотификации | Минимизира необходимостта от postMessage при всяко отваряне на приложението |

## Известни ограничения

- **PWA нотификации**: Работят само докато браузърът е отворен поради ограничение на Web APIs (Notification Triggers API не е широко поддържан). За надеждни нотификации в Android — използвайте APK.
- **iOS**: Нотификациите от PWA не се поддържат в Safari на iOS под версия 16.4. APK/Capacitor е единственото решение за iOS.
- **Офлайн режим**: Ако устройството е офлайн при `init()`, backend sync се пропуска мълчаливо и се използва последната кешираната конфигурация.
