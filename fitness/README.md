# KA-TRAINER — AI генератор на индивидуални тренировъчни планове

Самостоятелен, завършен модул: лендинг страница → въпросник (14 въпроса) →
AI генерира седмичен тренировъчен план с демонстрации, алтернативи и
адаптация на натоварването → AI персонален треньор в чат.

Работи независимо от основния NutriPlan, но е проектиран за бъдещо
свързване с него (виж „Интеграция с NutriPlan“ по-долу).

### Потокът на клиента

1. **Лендинг** (`index.html`) → CTA „Създай моя план“.
2. **Въпросник** (`app.html`): до 15 стъпки (жени виждат отделен въпрос за женско здраве),
   валидация на всяка стъпка, прогрес бар. Черновата се пази в localStorage
   (рестарт на страницата не губи отговорите).
3. **Генерация**: една POST заявка. Worker-ът: компактира профила → извлича
   релевантните насоки (mini-RAG) → 1 AI заявка → валидира JSON → локален
   matching на всяко упражнение към базата → закача GIF/thumbnail/инструкции
   + до 3 прекомпютнати алтернативи на упражнение → съхранява в KV → връща.
4. **План**: 7 дни с табове, карти на упражнения (снимка, серии×повторения,
   почивка, RPE, темпо, бележки), загрявка/разпускане, насоки, lightbox с GIF
   демонстрация (затваряне с ✕ / клик извън / Escape).
5. **Адаптация** — изцяло клиентска, 0 заявки:
   - **Замяна** — алтернативите идват с плана; изборът се пази локално.
   - **Олекотяване/затежняване** — детерминистични правила (серии ±1,
     почивка ∓, RPE бележка) върху базовия план; нивото се пази локално.
6. **AI треньор**: плаващ чат. Изпраща само `planId` + последните ≤6 реплики;
   Worker-ът зарежда компактния контекст (1 KV read) и прави 1 AI заявка.

### Matching: canonicalName + fuzzy token overlap

AI-ят връща за всяко упражнение `displayName` (БГ, за клиента) и
`canonicalName` (стандартно EN каталожно име) + `equipmentHint`/`bodyPart`.

1. Нормализация: долен регистър, без диакритика/пунктуация → tokens.
2. `score = съвпадащи думи / max(думи в заявката, думи в записа)`
3. Бонус при съвпадащ `equipmentHint` (+0.15) / `bodyPart` (+0.10) —
   разграничава „Bench Press“ с щанга от варианта с дъмбели.
4. Праг **0.35**; под него → fallback по категория (същото
   equipment/target), за да не остане празен слот.

Логиката е тествана в `test/matching.test.mjs` (24 теста, `npm test`).

## Икономичност (заложена в дизайна)

| Действие на клиента                  | AI заявки | Backend заявки | Бележка |
|--------------------------------------|:---------:|:--------------:|---------|
| Отваряне на лендинг/приложение        | 0         | 0              | статични файлове |
| Попълване на въпросника               | 0         | 0              | localStorage чернова |
| **Генерация на план**                 | **1**     | 1              | единствената „тежка“ заявка |
| Повторно отваряне на плана            | 0         | 0              | localStorage |
| Замяна на упражнение                  | 0         | 0              | прекомпютнати алтернативи |
| Олекотяване / затежняване             | 0         | 0              | детерминистични правила |
| GIF демонстрация (lightbox)           | 0         | 0              | jsDelivr CDN, lazy |
| Съобщение до AI треньора              | 1 (малка) | 1              | ~600 токена контекст, таван на изхода |

Допълнителни предпазители:

- **Rate limits в KV**: генерации 3/ден/IP, чат 30/ден/план (конфигурируеми
  през `GEN_DAILY_LIMIT` / `CHAT_DAILY_LIMIT`).
- **Exercise dataset**: 1 външен fetch на ~30 дни (KV TTL), не по заявка.
- **Чат история**: живее в браузъра; бекендът не пише нищо в KV за чата.
- **Compact coach context**: твърд таван ~3200 знака → предвидим разход.
- **GIF-овете се зареждат чак в lightbox-а**; в списъка са само статични
  thumbnails с `loading="lazy"`.

## Деплой

```bash
cd fitness

# 1. KV namespace (еднократно) — постави полученото id в wrangler.toml
npx wrangler kv namespace create FITNESS_KV

# 2. Secrets
npx wrangler secret put GEMINI_API_KEY --env production
npx wrangler secret put OPENAI_API_KEY --env production   # по избор (fallback)

# 3. Деплой
npm run deploy            # = wrangler deploy --env production

# 4. Фронтенд: статичните файлове се сервират както останалата част от
#    репото (GitHub Pages). URL на Worker-а по подразбиране:
#    https://aidiet-fitness.radilov-k.workers.dev
#    Override без редакция на кода: app.html?api=https://друг-url
```

По избор — предварително зареждане на индекса в KV (нулев cold-start fetch):

```bash
node scripts/build-exercise-index.mjs
npx wrangler kv key put exidx:v2 --path data/exercise-index.json \
    --binding FITNESS_KV --env production --remote
```

Локална разработка: `npm run dev` (worker на localhost:8787) +
`npm run serve` (статика на localhost:8080), после
`http://localhost:8080/app.html?api=http://localhost:8787`.

## PWA / app-like поведение

- **`manifest.json`** — приложението е инсталируемо (Android „Add to Home
  Screen“, iOS „Добави към начален екран“): standalone режим, портретна
  ориентация, shortcuts „Моята програма“ и „Нов план“.
- **`fitplan-sw.js`** — service worker (само `/fitness/` scope, регистрира се от
  `app.html`): онлайн винаги зарежда от мрежата (`cache: no-store`), офлайн
  fallback към кеш. CDN медия (thumbnail/GIF) е *cache-first* с таван 300.
  **Главният NutriPlan `sw.js` (корен `/`) не прехваща `/fitness/`** — иначе
  stale-while-revalidate връщаше остарели CSS при refresh на landing.
- **Landing (`index.html`)** не регистрира SW; при зареждане почиства
  отровени `/fitness/` записи от `nutriplan-*` кеша.
- **Системен „назад“** (Android бутон/жест, iOS swipe) се обработва
  in-app: затваря lightbox/чат, връща предишния въпрос във визарда,
  връща от плана към началния екран — без да изхвърля потребителя.
  Използва се „single sentinel“ модел: най-много един запис в историята,
  затова изходът от приложението никога не се блокира.
- **Mobile-first детайли**: всички полета са ≥16px (без iOS auto-zoom),
  touch targets ≥40–44px, `overscroll-behavior: contain` на скролируемите
  панели, safe-area отстъпи (dynamic island/жестова лента), тъмна
  `color-scheme` за нативните контроли.
- **Нулево мигане при зареждане**: критичен inline CSS (тъмен фон от
  първия байт), брандиран boot splash на app.html (маха се след първия
  кадър; авто-скрива се след 6 сек дори без JS), Google Fonts се зареждат
  асинхронно с `display=optional` (не блокират първия кадър и никога не
  правят видим FOUT "скок"), entrance анимациите не се пускат при
  първоначален рендер и при пре-рендер (интензитет/замяна), а видимите
  при зареждане reveal елементи на лендинга се показват мигновено.
  При ъпдейт на service worker-а страницата се презарежда еднократно
  (`controllerchange`), за да не останат смесени стари/нови файлове —
  точно това чупеше визията при старата cache-first стратегия.

## Интеграция с NutriPlan (бъдеща)

Модулът е самостоятелен, но границите са прокарани нарочно:

- **`clientRef`** — `POST /api/plan/generate` приема опционално
  `{ clientRef: "<nutriplan userId>" }`, което се съхранява при плана.
  NutriPlan може по-късно да намира тренировъчните планове на клиента.
- **`GET /api/plan/:id`** — планът е достъпен по id отвсякъде (същият формат,
  JSON), т.е. NutriPlan админ панелът може да го визуализира директно.
- **Общи конвенции** — същите env имена (`GEMINI_API_KEY`, `OPENAI_API_KEY`),
  същият модел (gemini-2.5-flash), същият стил Worker + KV, както основния
  `worker.js`. Сливането по-късно е механично: качване на handlers + KV ключове.
- **Хранене** — промптът на плана нарочно дава само 1-2 общи изречения за
  хранене („подробният хранителен режим е отделна услуга“) — мястото на
  NutriPlan не се дублира.
- **Профилът от въпросника** (`buildAnswers()` формата) е супермножество на
  това, което AI-ят ползва — може да се препрати към NutriPlan onboarding
  без повторно питане на клиента.
